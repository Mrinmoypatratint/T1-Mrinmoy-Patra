# SETUP.md

This document defines a **reproducible backend setup** for the Quiz Portal and captures the security/access-control expectations used for evaluation.

## 1) Scope

Backend stack:
- Django 6 + Django REST Framework
- SQLite (default, reproducible local DB)
- Firebase Auth token validation (server-side)

This backend is responsible for:
- quiz CRUD (with admin restrictions)
- attempt submission
- score calculation server-side
- result history retrieval

## 2) Prerequisites

- Python 3.11+
- Node.js 18+ (for frontend calls to backend)
- Firebase project with Google Sign-In enabled in Firebase Auth
- A Firebase Web API key (from project settings)

## 3) Repository setup

From project root:

python -m venv .venv
.venv\Scripts\activate
pip install -r apps/backend/requirements.txt

## 4) Environment configuration (required)

Create a backend environment file at `apps/backend/.env` (or export equivalent variables in your host environment):

```
DJANGO_SECRET_KEY=replace-with-a-long-random-secret
DJANGO_DEBUG=false
ALLOWED_HOSTS=localhost,127.0.0.1
CORS_ALLOWED_ORIGINS=http://localhost:5173,http://localhost:5174
FIREBASE_API_KEY=your-firebase-web-api-key
ADMIN_EMAILS=admin@example.com
```

### Variable meaning

- `DJANGO_SECRET_KEY`: required for Django cryptographic signing
- `DJANGO_DEBUG`: set `false` outside local dev
- `ALLOWED_HOSTS`: must be explicit; do not use wildcard `*` in production
- `CORS_ALLOWED_ORIGINS`: whitelist only real frontend origins
- `FIREBASE_API_KEY`: used server-side to validate Firebase ID tokens
- `ADMIN_EMAILS`: comma-separated admin allowlist (role enforcement source)

## 5) Database setup (clear, reproducible)

Default DB is SQLite at:
- `apps/backend/db.sqlite3`

Run migrations:

cd apps/backend
python manage.py migrate

Create migration files if model changes are introduced:

python manage.py makemigrations quizzes
python manage.py migrate

### Reset local DB safely (development only)

If you need a clean local state:

1. Stop backend server
2. Delete `apps/backend/db.sqlite3`
3. Re-run `python manage.py migrate`

> Do not use this reset procedure in production.

## 6) Run backend

From `apps/backend`:

python manage.py runserver 0.0.0.0:8000

Health endpoint:
- `GET /api/health`

## 7) Required API contracts (as evaluated)

- `POST /api/quizzes`
- `GET /api/quizzes`
- `GET /api/quizzes/:id`
- `POST /api/attempt`
- `GET /api/results/:userId`

Implemented in:
- `apps/backend/quizzes/urls.py`
- `apps/backend/quizzes/views.py`

## 8) Security & validation controls (production-grade expectations)

### 8.1 Token validation (server-side)

Every protected endpoint extracts `Authorization: Bearer <idToken>` and validates the token against Firebase.

Code reference:
- `apps/backend/quizzes/authentication.py`
- `_get_auth_user(...)` in `apps/backend/quizzes/views.py`

If token is missing/invalid, backend returns `401`.

### 8.2 Access control

- Admin-only operations (`POST/PATCH/DELETE` quiz management) are enforced on the backend.
- Admin role is derived from `ADMIN_EMAILS` environment variable.
- Users can only read their own results unless admin.

### 8.3 Server-side grading (don’t trust client)

- Client submits only selected answers.
- Backend loads authoritative correct answers from DB.
- Score is computed server-side in `POST /api/attempt`.
- `correctAnswerIndex` is not returned by quiz-fetch endpoint.

### 8.4 Input validation

Backend validates:
- quiz required fields (`title`, `timeLimit`, `questions`)
- `timeLimit` positive integer
- each question has exactly 4 options
- `correctAnswerIndex` in range `0..3`
- duplicate attempts blocked per user+quiz

## 9) Reproducibility checklist

A setup is correct when all are true:

- `python manage.py check` reports no issues
- `python manage.py migrate` completes successfully
- `GET /api/health` returns 200
- unauthenticated call to `GET /api/quizzes` returns 401
- authenticated user can list quizzes
- score from `POST /api/attempt` is generated by backend logic

## 10) Common misconfigurations to avoid

- Leaving `DJANGO_DEBUG=true` in deployed environments
- Using permissive CORS (`*`) in production
- Missing `FIREBASE_API_KEY` (causes token verification failures)
- Treating frontend role flags as trusted authority
- exposing answer keys in quiz read APIs
